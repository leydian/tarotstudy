# Persona Contract

아르카나 성소의 유지보수 의사결정을 위한 운영 계약서입니다.

## 1. 운영 우선순위
1. 사용자 체감 품질
2. 안정성
3. 개발 편의

## 2. 페르소나 정의

### 개발자 페르소나: 신뢰성 엔지니어
- 목표: 장애와 회귀 없이 예측 가능한 변경
- 책임 범위:
  - `/api/reading` 계약 무결성 유지
  - fallback 경로 및 예외 처리 보장
  - 타입 안정성과 디버깅 가능성 확보
- 승인 기준:
  - 필수 응답 필드 누락 없음
  - 빌드/검증 스크립트 통과

### 기획자 페르소나: 품질 오케스트레이터
- 목표: 질문 의도별 일관된 품질 제공
- 책임 범위:
  - 질문 유형(light/deep/binary/relationship/career/emotional) 기준 정의
  - 톤/서사/행동지침 품질 점검
  - 릴리스 우선순위 관리
- 승인 기준:
  - 질문 유형별 체감 변화 설명 가능
  - 품질 게이트 위반 없음

### 사용자 페르소나: 결정 보조 탐색자
- 목표: 공감되는 해석과 실행 가능한 조언 확보
- 기대 경험:
  - 질문 후 빠른 결과 노출
  - 이해 가능한 결론과 행동지침
  - 불필요한 기술 용어 최소화
- 관측 지표:
  - 결과 노출 비율
  - report/study 탭 전환 비율
  - 새 질문 시작 전환율

## 3. 의사결정 규칙
- 충돌 시 개발 편의보다 사용자 체감 품질을 우선합니다.
- 품질 저하 우려가 있는 변경은 기능 추가보다 릴리스 우선순위를 낮춥니다.
- 회귀 가능성이 큰 변경은 반드시 회귀 시나리오 검증을 통과해야 합니다.

## 4. PR 승인 규칙
- PR 본문에 아래 3가지 관점을 모두 작성해야 합니다.
  - 개발자 관점 영향
  - 기획자 관점 품질 변화
  - 사용자 관점 체감 변화
- 하나라도 누락되면 머지하지 않습니다.

## 5. 변경 관리
- 응답 계약 변경은 타입/문서/테스트를 같은 PR에서 동기화합니다.
- fallback 관련 로직 변경 시 `fallbackReason` 기록 경로를 유지합니다.
