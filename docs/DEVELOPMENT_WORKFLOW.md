# Development Workflow Standard

웹 앱 유지보수 시 AGENTS/CLAUDE 원칙을 실무 절차로 고정한 실행 기준입니다.

## 1. 작업 시작 규칙 (Before Coding)
- 작업 시작 전에 아래 4가지를 명시합니다.
  - 대상: 수정할 레포/경로/파일
  - 작업 유형: 조사/구현/리뷰/문서
  - 산출물: 코드 패치/리포트/테스트 결과
  - 성공 기준: 통과해야 할 테스트/검증 명령
- 요구사항이 모호하면 먼저 로컬 파일/코드에서 확인 가능한 사실을 수집합니다.

## 2. 구현 규칙 (During Coding)
- 요청 범위를 벗어난 기능 추가를 금지합니다.
- 변경은 국소적으로 적용합니다. 관련 없는 리팩터링은 하지 않습니다.
- 기존 코드 스타일과 계약(API/타입)을 유지합니다.
- 실패 시 고정 루프를 따릅니다.
  1. 실패 증상 요약
  2. 원인 가설 1~2개
  3. 최소 비파괴 검증
  4. 최소 수정
  5. 재검증

## 3. 검증 규칙 (Verification)
- 백엔드/프론트 변경 시 기본 검증:
  - `npm run test:persona --prefix apps/api`
  - `npm run build --prefix apps/web`
- 네트워크/외부 API 관련 변경 시 추가 검증:
  - 응답의 `apiUsed`, `fallbackUsed`, `meta.fallbackReason` 확인
  - 실패 로그에서 `status`, `cause`, `timed_out` 확인

## 4. 문서 동기화 규칙 (Docs Freshness)
- 코드 계약/운영 동작을 변경하면 같은 작업에서 문서를 함께 갱신합니다.
  - `docs/CHANGELOG.md`
  - 필요한 경우 `docs/RELEASE_NOTES_*.md`
  - 설정 기본값 변경 시 `docs/SETUP_SECURITY.md`

## 5. PR 작성 규칙
- `.github/pull_request_template.md` 체크리스트를 모두 채웁니다.
- PR 본문에는 아래를 필수로 포함합니다.
  - 변경 목적과 범위
  - 검증 명령과 결과
  - 리스크/롤백 포인트

## 6. 금지 항목
- 근거 없이 추정으로 구현하지 않습니다.
- 같은 실패 명령을 가설 수정 없이 반복하지 않습니다.
- 무관한 코드 정리/포맷 변경을 끼워 넣지 않습니다.

## 7. 웹 앱 전용 체크포인트
- 질문 의도와 스프레드 선택이 일치하는지 확인합니다.
  - 예: 일상 양자택일 질문에서 커리어형 문맥이 노출되지 않아야 함
- 결과 화면에서 사용자 체감 품질을 확인합니다.
  - 결론/지침의 톤이 질문 무게와 일치하는지
  - fallback 시에도 응답 중단 없이 결과가 노출되는지

## 8. 완료 정의 (Definition of Done)
아래를 모두 만족하면 완료로 간주합니다.
- 구현 범위가 요청과 일치함
- 필수 검증 통과
- 관련 문서 동기화 완료
- 변경/검증/리스크를 보고에 명시함
